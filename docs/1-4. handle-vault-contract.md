# Typed Handle and Vault Contract

> **쉽게 말하면:** Vault는 민감한 원문을 넣는 폴더가 아니라, 암호화된 데이터와 그 데이터를
> 가리키는 보관증(handle)을 연결하는 내부 금고다. Agent는 보관증만 받고 금고의 위치나 내용을
> 직접 볼 수 없다. 용어는 [`glossary.md`](glossary.md)를 참고한다.

## 기본 결정

Agent와 LLM에는 ciphertext와 Vault 위치 대신 opaque, typed, session-bound handle을 전달한다.
순차 번호나 사용자가 직접 만들 수 있는 marker는 권한 capability로 인정하지 않는다.

## Handle metadata

Core 내부 handle record는 최소 다음 정보를 가진다.

```text
handle_id
session_id
kind
value_type
key_id
context_id
shape / slot_count
origin_request_id
provenance
allowed_operations
reveal_class
created_at
expires_at
single_use
status
```

- `handle_id`는 CSPRNG로 생성한 예측 불가능한 값으로 한다.
- handle은 하나의 session, key와 context에 바인딩한다.
- marker 문자열은 handle의 표시 형식일 뿐 신뢰 근거가 아니다.
- Core record가 없거나 metadata가 일치하지 않으면 거부한다.

## Handle 종류

| 종류 | 의미 | Agent 권한 |
|---|---|---|
| input secret handle | 사용자 원문에서 생성된 비밀 | 허용된 연산 참조만 가능 |
| computed result handle | 검증된 연산 결과 | 후속 연산 또는 reveal policy 입력 |
| reveal request handle | egress policy가 만든 요청 | Agent가 직접 만들 수 없음 |
| public value | 정책상 공개된 상수/metadata | 허용 범위에서 평문 사용 가능 |

Input secret handle은 Agent 응답에 등장했다는 이유만으로 복호하지 않는다. 기본 reveal 대상은
검증된 provenance를 가진 computed result handle이다.

## Marker 처리

- 사용자 원문에 marker와 비슷한 문자열이 있으면 escape 또는 일반 텍스트로 처리한다.
- Agent가 새 marker 문자열을 만들어도 Core record가 없으면 무효다.
- 다른 세션 marker와 handle은 사용하지 못한다.
- marker type과 Core record type이 다르면 거부한다.
- unknown marker를 원문 그대로 출력하거나 자동 복호하지 않는다.
- 문자열 치환 전 내부적으로 구조화된 segment/handle 표현을 유지한다.

## Vault 규칙

- Vault는 Agent sandbox에 mount하지 않는다.
- Agent와 LLM에 Vault path, record index 또는 전체 목록을 제공하지 않는다.
- ciphertext와 metadata 저장은 result handle 반환보다 먼저 완료한다.
- compute, result 저장과 handle 발급은 하나의 논리적 transaction으로 처리한다.
- partial write 또는 저장 실패 시 handle을 발급하지 않는다.
- 세션 종료 시 ephemeral handle과 ciphertext를 폐기한다.

## 저장 형식

초기 버전은 **메모리 기반 Vault**를 사용한다. 별도의 DB나 JSON 파일을 만들지 않으며 프로세스 또는
세션이 끝나면 해당 세션의 record를 폐기한다. 이 단계에서는 재시작 후 복구를 지원하지 않는다.

영구 저장이 필요해지는 후속 버전의 기본안은 **SQLite + binary BLOB**이다.

| 저장 대상 | 권장 형식 | 쉬운 설명 |
|---|---|---|
| handle, session, 만료 시각, scheme과 policy | SQLite의 형식화된 column | 금고의 서랍 목록과 사용 규칙 |
| CKKS, BFV/BGV, Boolean FHE ciphertext | backend가 직렬화한 binary BLOB | FHE로 잠긴 봉투 |
| exact secret | AEAD ciphertext binary BLOB | 주민등록번호처럼 그대로 복원할 값을 담은 잠긴 봉투 |
| exact secret의 DEK | 2-of-2 threshold-wrapped binary BLOB | PC와 스마트폰이 함께 있어야 여는 봉투의 열쇠 |
| nonce, context/key-set version, 무결성 metadata | 형식화된 column 또는 binary BLOB | 잘못된 열쇠나 변조를 찾기 위한 꼬리표 |

JSON은 사람이 확인하는 설정이나 제한된 진단 내보내기에만 사용한다. 암호문을 JSON에 넣으려면
Base64 변환이 필요해 크기가 커지고, metadata와 암호문을 하나의 transaction으로 저장하기도 어렵다.
`profile.json`은 암호화 설정 자료이지 Vault 본체가 아니다.

SQLite를 채택하더라도 DB 파일 전체를 안전하다고 가정하지 않는다. ciphertext 외 metadata도 사용
패턴을 노출할 수 있으므로 최소한만 저장하고, host 권한으로 접근을 제한하며, crash recovery와 DB
파일 암호화는 별도의 threat model과 검증을 거친 뒤 활성화한다.

## 기본 제한

- raw ciphertext Base64를 Agent에 기본 반환하지 않는다.
- handle enumeration API를 제공하지 않는다.
- cross-session clone과 import를 제공하지 않는다.
- 무제한 TTL을 사용하지 않는다.
- handle만으로 인증을 대체하지 않는다.
- operation count와 reveal count를 handle provenance에 기록한다.

## 추후 검토

- handle entropy와 wire encoding
- 세션 TTL, result TTL과 one-time 사용 기준
- 영속 profile handle과 ephemeral conversation handle 분리
- SQLite schema, DB 파일 암호화, encrypted journal과 crash recovery
- provenance DAG의 최소 필드
- garbage collection과 secure deletion 범위
