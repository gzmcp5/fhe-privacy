# Pre-LLM Ingress Security Boundary

이 문서는 RAG 없는 FHE-Privacy 아키텍처의 첫 번째 확정 경계인 pre-LLM ingress를 정의한다.
Hermes Agent와 LLM은 비신뢰 영역이며, OpenShell은 Hermes 전체 프로세스의 실행 격리 계층으로 사용한다.

## 결정

> Secure Gateway는 사용자 원문의 유일한 진입점이다. OpenShell sandbox의 Hermes Agent에는
> Gateway가 마스킹에 성공한 메시지만 전달한다.

- 사용자 원문은 신뢰 호스트의 Secure Gateway만 수신한다.
- Hermes Agent 전체는 OpenShell sandbox 안에서 실행한다.
- Agent와 LLM에는 원문 대신 marker가 포함된 masked envelope만 전달한다.
- Agent가 선택적으로 호출하는 MCP masking tool은 secure-gateway 경계로 인정하지 않는다.
- 마스킹, 암호화 또는 Vault 저장이 실패하면 Agent를 호출하지 않는다.
- Gateway를 강제할 수 없는 Agent 연결은 secure-gateway 모드로 표시하지 않는다.

## 신뢰 경계

| 영역 | 구성요소 | 허용 데이터 |
|---|---|---|
| 신뢰 호스트 ingress | 사용자 입력 adapter, Secure Gateway, PII policy, crypto ingress, Vault | 사용자 원문, marker, ciphertext |
| 비신뢰 OpenShell sandbox | 외부 Agent, Agent MCP bridge | masked text, opaque handle, 공개 metadata |
| 비신뢰 외부 서비스 | LLM provider | masked prompt와 허용된 공개 정보 |

OpenShell sandbox에는 secret key, Vault 파일, reveal executable, Gateway credential 또는 사용자
홈 디렉터리를 노출하지 않는다. OpenShell은 전체 Agent 프로세스를 격리해야 하며 Agent가 실행하는
일부 shell command만 격리하는 구성은 이 경계를 만족하지 않는다.

## 정상 흐름

1. 신뢰 ingress adapter가 사용자 원문을 받는다.
2. Gateway가 입력을 정규화하고 PII policy를 실행한다.
3. 민감값을 암호화하고 Vault에 저장한다.
4. Gateway가 opaque marker를 발급하고 원문 span을 marker로 교체한다.
5. Gateway가 원문이 제거됐는지 검증한다.
6. 검증에 성공한 masked envelope만 OpenShell Hermes Agent에 전달한다.
7. Agent는 masked prompt만 LLM에 전달한다.

## Masked envelope

내부 메시지는 최소한 다음 정보를 가진다.

```json
{
  "sessionId": "opaque-session-id",
  "messageId": "opaque-message-id",
  "maskedText": "예산은 {{amount:opaque-capability}}원입니다",
  "policyVersion": "1"
}
```

Agent에는 원문, 원래 span, Vault 경로, plaintext hash, secret key 식별 정보, reveal credential을
전달하지 않는다. Marker와 handle의 상세 수명주기 및 권한 계약은 후속 Vault 설계에서 확정한다.

## Fail-closed 규칙

| 실패 또는 불확실성 | 동작 |
|---|---|
| PII detector 예외 | 전송 중단 |
| 고위험 ambiguity | 사용자 확인 또는 전송 중단 |
| 암호화 실패 | 전송 중단 |
| Vault 저장 실패 | 전송 중단 |
| masking 후 원문 잔존 | 전송 중단 |
| 지원하지 않는 attachment 또는 입력 channel | secure-gateway 모드에서 거부 |
| Agent adapter 연결 실패 | 원문 fallback 금지 |
| OpenShell 격리 검증 실패 | secure-gateway 실행 금지 |
| 정책 버전 불일치 | 세션 중단 |

오류 처리에는 원문을 Agent로 전달하는 fallback을 두지 않는다.

## 초기 지원 범위

초기 secure-gateway 모드가 지원하는 입력은 Gateway가 직접 수신한 UTF-8 텍스트로 제한한다.

초기 비지원 입력:

- 파일 attachment
- 이미지와 OCR
- 음성 입력
- 외부 memory import
- 임의 workspace 파일의 prompt 삽입
- 다른 MCP 서버가 반환한 민감한 평문
- Agent 간 평문 메시지

비지원 입력은 조용히 통과시키지 않고 거부한다. 지원 범위가 늘어날 때마다 동일한 pre-LLM
경계와 누출 회귀 테스트를 추가한다.

## OpenShell 강제 조건

- 전체 Agent 프로세스를 sandbox 내부에서 실행한다.
- filesystem 격리는 사용할 수 없으면 시작을 중단하는 hard requirement로 설정한다.
- Agent에는 secret이 없는 전용 workspace만 제공한다.
- FHE-Privacy key, Vault, profile, IPC credential 경로를 mount하지 않는다.
- 직접 LLM provider 접속을 허용하지 않고 승인된 inference route만 사용한다.
- 불필요한 network endpoint와 host IPC 접근을 거부한다.
- OpenShell log와 Agent transcript에는 Gateway가 만든 masked envelope 이후의 데이터만 들어간다.

OpenShell은 Agent 실행을 격리하지만 PII masking과 Gateway 강제를 대신하지 않는다.

## 완료 검증

pre-LLM ingress가 완료되려면 다음 자동 검증이 모두 통과해야 한다.

- 실행마다 생성한 canary secret이 Gateway 이후 Agent 입력에서 발견되지 않는다.
- canary secret이 mock LLM request, Agent transcript, MCP stderr, OpenShell log에 없다.
- Agent 입력과 mock LLM request에는 대응 marker만 존재한다.
- detector, 암호화, Vault 오류 시 Agent 호출 횟수가 0이다.
- 지원하지 않는 attachment와 입력 channel이 거부된다.
- Agent가 Gateway를 우회해 원문을 받는 실행 경로가 없다.
- OpenShell sandbox에서 key와 Vault 경로를 읽을 수 없다.
- Agent의 직접 provider 연결이 차단된다.

## 후속 설계와의 경계

이 결정은 pre-LLM ingress만 확정한다. 다음 항목은 별도 단계에서 다룬다.

- stdio MCP bridge와 stateful Privacy Core의 프로세스 배치
- Agent, Gateway, Reveal Authority의 권한 채널 분리
- marker와 handle의 session binding, TTL, provenance
- post-LLM terminal egress와 reveal policy
- CKKS 대상과 exact secret 저장 방식
