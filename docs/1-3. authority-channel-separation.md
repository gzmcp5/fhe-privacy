# Authority and Channel Separation

## 기본 결정

Agent-safe 연산, Gateway ingress, local reveal을 서로 다른 권한 채널로 분리한다. 호출자가 제출한
`role` 값은 보안 경계로 사용하지 않는다.

## 권한 영역

| 영역 | 보유 material | 허용 기능 |
|---|---|---|
| OpenShell Agent 영역 | session-scoped agent capability | handle 기반 공개 연산 요청 |
| Agent MCP Bridge | 독자 권한 없음 | stdio MCP와 agent-safe Core protocol 중계 |
| Public Compute Worker | public context, public/evaluation key | secret-free 동형 연산 |
| Gateway Ingress | 사용자 원문, joint public material | 검출, 마스킹, FHE/AEAD 암호화, 세션 생성 |
| Vault Coordinator | handle metadata, ciphertext | lookup, 저장, provenance 검증 |
| Reveal Coordinator | policy, destination, nonce, transcript | 승인 및 partial decrypt protocol 조정; secret share 없음 |
| PC Partial Decrypt Authority | `sk_pc`, exact-envelope PC share | 승인된 partial decrypt/unwrap share 생성 |
| Phone Partial Decrypt Authority | `sk_phone`, exact-envelope phone share | 사용자 승인 후 partial decrypt/unwrap share 생성 |
| PC/Phone Fusion Sink | 승인된 두 partial share | 지정된 장치에서만 plaintext 생성 |

## 강제 규칙

- Agent-facing 프로세스에는 decrypt/reveal 구현과 secret key loader를 포함하지 않는다.
- Agent MCP Bridge와 Hermes는 같은 비신뢰 권한 영역으로 취급한다. Bridge credential의 기밀성이나
  binary identity를 authorization 근거로 사용하지 않는다.
- Public Compute Worker는 secret key 경로를 mount하거나 import하지 않는다.
- Gateway가 사용하는 host-only interface와 Agent가 사용하는 interface를 분리한다.
- Reveal Coordinator, partial authority와 fusion sink는 일반 MCP tool registry에 등록하지 않는다.
- Agent capability로 host-only 또는 reveal interface에 접근할 수 없다.
- channel identity와 scope는 Core가 발급하고 검증한다.
- agent capability가 Hermes에 노출돼도 agent-safe operation 밖으로 권한이 확대되지 않아야 한다.
- 하나의 wildcard scope 또는 `all` role을 제공하지 않는다.
- 권한 상승은 자동 fallback 없이 명시적 정책/사용자 승인 경로만 허용한다.
- 완성된 FHE secret key나 단일 exact-secret master key를 생성, 저장, 전송 또는 recovery artifact에 넣지 않는다.
- partial share는 `session_id`, `request_id`, ciphertext hash, context/key-set, destination, nonce, expiry와
  policy decision에 인증 방식으로 바인딩한다.

## 프로세스 기본값

초기 구현에서도 Agent MCP Bridge, Privacy Core, Public Compute Worker, Reveal Coordinator와 PC
Partial Decrypt Authority를 별도 프로세스와 권한 채널로 둔다. Phone Partial Decrypt Authority는
스마트폰 앱의 별도 trust boundary에 둔다. 단일 프로세스 spike는 폐기 가능한 검증용으로만 표시한다.

```text
OpenShell Hermes Agent
  -> Agent MCP Bridge
  -> HTTPS/mTLS agent-safe Core channel
  -> Public Compute Worker

Secure Gateway
  -> local host-only UDS
  -> Gateway Ingress / Vault Coordinator

Gateway Egress
  -> local Reveal Coordinator UDS
  -> PC Partial Decrypt Authority + Smartphone Partial Decrypt Authority
  -> selected PC Fusion Sink or Phone Fusion Sink
```

## 거부 규칙

- Agent가 raw plaintext를 FHE operation 인자로 제출하면 거부한다.
- Agent가 decrypt, resolve, export-key 또는 list-vault를 요청하면 거부한다.
- Agent-safe channel에 reveal handle 생성을 직접 요청하는 범용 API를 두지 않는다.
- 연결 scope와 session이 일치하지 않으면 거부한다.
- channel credential이 누락, 만료 또는 폐기됐으면 거부한다.
- 어느 한쪽 partial share, 사용자 승인 또는 destination binding이 없으면 reveal을 거부한다.
- single-key, PC-only 또는 phone-only decrypt fallback을 제공하지 않는다.

## Public Compute 신뢰 범위

Public Compute Worker는 plaintext와 secret key를 맡기지 않는다는 의미에서 confidentiality-untrusted
영역으로 취급한다. 따라서 Worker 침해가 곧바로 plaintext 복구로 이어지지 않아야 한다. 반면 악성
Worker는 잘못된 ciphertext, 재생 결과 또는 자원 고갈을 만들 수 있으므로 결과 provenance와
context/session 결합을 Core가 검증해야 한다. 그러나 provenance metadata만으로 실제 연산의 정확성을
증명할 수는 없다. 초기 버전은 Worker를 기밀성 측면에서는 비신뢰, 결과 무결성 측면에서는 신뢰된
로컬 프로세스로 제한한다. 악성 Worker까지 지원하려면 중복 계산, 검증 가능한 연산 또는 별도 proof
체계를 도입한다. FHE 자체가 연산자의 정직성을 보장한다고 주장하지 않는다.

## 추후 검토

- macOS/Linux/Windows별 local UDS/named pipe와 OS identity
- agent-safe HTTPS endpoint의 certificate lifecycle과 Kubernetes Service 배치
- Public Compute Worker의 별도 local process와 별도 service 배포 선택 기준
- PC/phone share의 hardware-backed storage, enrollment, rotation과 2-of-3 recovery
- capability 발급과 회수 protocol
