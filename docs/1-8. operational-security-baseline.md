# Operational Security Baseline

## 기본 결정

OpenShell을 비신뢰 Agent의 기본 격리 backend로 사용하고, 격리·로그·resource policy를 확인할 수
없으면 secure-gateway 모드 실행을 중단한다.

## OpenShell 기본 제한

- 전체 Hermes Agent 프로세스와 Agent MCP Bridge를 sandbox 안에서 실행한다.
- 일부 shell command만 sandboxing하는 구성은 secure-gateway로 인정하지 않는다.
- filesystem enforcement는 지원되지 않으면 시작을 중단하는 hard requirement로 설정한다.
- Agent는 비특권 user/group으로 실행한다.
- secret이 없는 전용 workspace만 read/write로 허용한다.
- key, Vault, profile, host home과 reveal credential 경로를 mount하지 않는다.
- 직접 provider 접속을 차단하고 승인된 inference route만 사용한다.
- network endpoint는 deny-by-default와 최소 allowlist를 적용한다.
- broad host bind, wildcard endpoint와 임의 localhost 접근을 허용하지 않는다.

## 로그와 관찰성

허용 로그:

- correlation ID
- session의 비밀이 아닌 식별자
- operation 종류
- 성공/거부/오류 코드
- 처리 시간과 payload 크기 범주
- policy version

기본 금지 로그:

- 사용자 원문
- 복호 plaintext
- exact secret
- secret key와 credential
- 전체 ciphertext/handle dump
- masked 전 입력을 포함한 stack trace

stdout은 MCP protocol 전용으로 유지하고 diagnostic log는 redacted stderr 또는 별도 audit sink로
보낸다. Crash report, test snapshot과 debug mode에도 같은 규칙을 적용한다.

## Resource 제한

- 사용자 입력, MCP message와 ciphertext payload 최대 크기
- 세션당 handle 수
- operation별 timeout과 cancellation
- multiplication depth와 연산 횟수
- 동시 작업 수와 memory quota
- reveal 요청 횟수
- 반복 실패와 brute-force handle 요청 rate limit

한도를 초과하면 부분 결과나 낮은 보안 fallback을 사용하지 않는다.

## 수명주기와 장애

- 초기 Privacy Core는 session-scoped로 실행한다.
- Core crash/restart 시 세션을 복구하려 하지 않고 종료한다.
- 세션 종료 시 ephemeral capability와 Vault state를 폐기한다.
- orphan ciphertext와 temp file 정리 정책을 둔다.
- key/profile backup은 별도 암호화와 접근 통제 없이 만들지 않는다.
- software update와 policy 변경 후 sandbox/Core compatibility를 재검증한다.

## 공급망과 배포

- OpenFHE wheel과 release artifact에 checksum/signature 검증을 추가한다.
- OpenShell과 Agent image/version을 기록하고 지원 조합을 명시한다.
- 외부 Agent plugin, MCP server와 adapter 설정 변경을 audit한다.
- one-click install은 실행 명령, mount, network 권한을 사용자에게 표시한다.
- OpenShell이 alpha인 동안 production 보안 보증을 주장하지 않는다.

## 검증 기준

- sandbox 내부에서 key/Vault 경로 읽기가 거부된다.
- 직접 provider와 미허용 network endpoint 연결이 거부된다.
- host-only/reveal interface 접근이 거부된다.
- canary secret이 Agent, LLM mock payload와 모든 log에서 발견되지 않는다.
- quota, timeout과 malformed request가 fail-closed로 처리된다.
- Core/Bridge crash가 plaintext fallback을 만들지 않는다.
- sandbox enforcement가 비활성화되면 secure-gateway 시작이 실패한다.

## 추후 검토

- 지원 OS와 OpenShell runtime 조합
- local Docker/Podman/VM 및 remote deployment 정책
- audit 보존 기간과 사용자 진단 UX
- clean-machine install 검증
- incident response와 key compromise 절차
- production readiness 조건

## 운영상 잔여 위험

OpenShell policy가 정상 적용돼도 sandbox escape와 host OS compromise 가능성은 남는다. 또한
Hermes/OpenShell/OpenFHE 및 패키지 공급망 오염, reveal 이후 화면·클립보드 탈취, payload
크기·처리 시간·operation pattern을 통한 metadata/timing leakage와 구현 버그는 별도 위협으로
관리한다. 전체 잔여 위험과 보안 주장 한계는
[`1-0. security-architecture-index.md`](1-0.%20security-architecture-index.md)를 기준으로 한다.
