# FHE-RAG 코퍼스 위탁(offload) 위협 노트

> **상태: 보류(deferred).** 이 문서는 암호화된 코퍼스를 비신뢰 인프라(Vector DB 호스트)에
> 위탁하는 **모델 2** 설계에서 나중에 반드시 다뤄야 할 위협을 기록해 둔 참고 노트다.
> 현재 진행에서는 **일단 무시하고**(= 접근 패턴 누출 감수, 무결성 후순위) 다이어그램/구현을
> 진행한다. 실배포/위협모델 확정 단계에서 이 문서를 다시 펼친다.

## 0. 배경 — 두 저장 모델

RAG 한 레코드 `{id, embedding, text}`를 어디에 두느냐로 갈린다.

| | 모델 1 | 모델 2 (현재 선택) |
|---|---|---|
| 임베딩 | 암호화(CKKS) 위탁 | 암호화(CKKS) 위탁 |
| 텍스트 | **로컬 평문** | **대칭암호화(AES 등)해서 위탁** |
| 로컬 저장 | 텍스트 전부 | 키만 |
| 접근 패턴 누출 | 없음 | **있음** |

- 조인 키는 **id**. `id → 암호화 임베딩`(검색용, 동형연산)과 `id → 텍스트`가 1:1 정합해야 한다.
- 임베딩은 **단방향**이라 텍스트로 복원 불가 → 텍스트는 "저장해뒀다 id로 조회"하는 것이지
  임베딩에서 만들어내는 게 아니다. 모델 2에서 새로 복호하는 건 **텍스트(대칭키)**이지
  임베딩이 아니다.

## 1. 접근 패턴 누출 (access-pattern leakage)

### 1.1 무엇인가

비신뢰 서버는 요청을 서비스하려면 **특정 슬롯(물리 주소)을 실제로 읽어서** 돌려줘야 한다.
따라서 내용·id 값·쿼리 내용이 모두 암호화돼 있어도, 서버는 **"어떤 슬롯을 접근했는가"**를
관찰한다.

- **핵심**: 누출의 원천은 "암호화된 id"가 아니라 **물리적 슬롯 접근**이다.
  id를 암호화해 고정 토큰으로 바꿔도 슬롯 접근 자체는 관찰되므로 못 막는다.
- **슬롯 위치 = 안정적인 가명(pseudonym) id**. "S1234가 뭔지"는 몰라도 "S1234가 또 읽혔다"는
  셀 수 있다.

### 1.2 두 가지 신호

- **빈도(frequency)**: 각 슬롯이 얼마나 자주 접근되는가.
- **동시 접근(co-access)** — 보통 이쪽이 더 강력:
  - 한 쿼리의 top-k를 함께 가져오면 → 그 k개 슬롯이 **서로 관련**있다는 신호.
  - 여러 쿼리에 걸쳐 같은 슬롯이 반복 등장 → 쿼리 간 상관.

```
쿼리1 → {S1234, S5678, S9012}
쿼리2 → {S1234, S5678, S3456}   ← S1234·S5678 공유 = 강한 상관 신호
```

### 1.3 왜 "쿼리 암호화"만으로 부족한가

접근 패턴이 **쿼리의 지문**이 된다. 서버는 내용을 몰라도 "쿼리 A형 vs B형"을 군집화할 수 있고,
그 가명 군집에 **의미를 연결하는 앵커가 하나만 생기면** 전체가 풀린다. 그 앵커가 보조지식이다.

### 1.4 실제 위협이 되는 조건 (auxiliary knowledge)

- **알려진/공개 코퍼스**: 진짜 문서들의 동시출현 구조를 계산해 관찰된 co-access 그래프와 매칭
  → 슬롯↔문서 역매핑 (IKK, count attack 등).
- **문서 주입(chosen-document / landmark)**: 공격자가 내용을 아는 문서를 심고, 그게 top-k에
  뜨는 쿼리를 그 주제로 특정.
- **쿼리 지문·반복**: "같은 쿼리 반복", "여러 사용자가 같은 검색", "특정 시각 급증" 등 행동 정보.
- **외부 이벤트 상관**: 접근 급증 시점을 실제 사건과 대조.

### 1.5 저위험 조건 (감수 타당)

아래가 **모두** 성립하면 실효 위협이 낮다:

- 코퍼스가 **비공개·고엔트로피**
- 공격자가 **문서/쿼리 주입 불가**
- 서버가 **honest-but-curious**(수동 관찰만)
- **"무엇을 검색하는지" 메타데이터 자체가 크게 민감하지 않음**

### 1.6 완화책 (ORAM 없이, 누출을 줄이지만 없애진 못함)

- **더미 슬롯 섞기(padding)**: 매 쿼리에 랜덤 슬롯 추가 → 빈도·co-access에 노이즈.
- **로컬 캐싱**: 한 번 가져온 청크 재사용 → 반복 접근 감소.
- **커버 쿼리**: 주기적 가짜 쿼리로 지문 흐리기.
- **완전 은닉**: PIR / ORAM — 서버가 "어느 걸 원하는지" 모르게. 비용이 크다(사실상 매 쿼리
  전체 스캔급). 트릴레마: **{로컬 저장 최소화 · 접근패턴 은닉 · 저렴함} 셋 중 둘만.**

## 2. 무결성 / 변조 (integrity / tampering)

### 2.1 위협

비신뢰 서버가 텍스트/임베딩 암호문을 **바꿔치기·삭제·stale 반환**할 수 있다.

- id 42를 요청했는데 다른 청크의 암호문을 반환(스왑) → 잘못된 컨텍스트가 프롬프트로.
- 청크를 조용히 드롭 → 검색 품질 저하 또는 fail.
- 오래된 버전 반환.

모델 1은 텍스트가 로컬이라 이 **원격 변조 위험이 없다**. 모델 2에서만 생기는 문제.

### 2.2 해결책 (나중에 값싸게 얹을 수 있음)

- **AEAD (AES-GCM 등)** 로 텍스트 암호화 — 기밀성 + 무결성.
- **암호문을 id에 바인딩**: `id`를 AAD(associated data)로 넣어 "이 암호문은 진짜 id 42의 것"을
  검증 → 스왑 탐지.
- **인덱스 전체 무결성**: 전체 매핑에 대한 서명/Merkle 등으로 드롭·리플레이 방어(더 무거움).
- 검증 실패 시 **fail-closed**.

## 3. 현재 결정(decision log)

- **접근 패턴 누출**: 감수하고 진행. 전제는 §1.5 저위험 조건(특히 비공개 코퍼스 +
  honest-but-curious 서버). 실배포 위협모델에서 재확인 필요.
- **무결성/변조**: 지금은 다루지 않음. 추후 §2.2(AEAD + id 바인딩)부터 적용.
- **텍스트 복호**: FHE(CKKS)가 아니라 **대칭키** 별도 경로. 다이어그램/구현에서 FHE Client와 구분.
