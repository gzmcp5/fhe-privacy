# Security Architecture Decision Index

이 문서는 RAG 없는 FHE-Privacy 재설계의 보안 결정 인덱스다. 외부 Agent와 LLM을 비신뢰
영역으로 두고 Hermes Agent, OpenShell과 FHE-Privacy를 함께 사용하는 구조를 기준으로 한다.

## 상태 의미

- **기본 채택**: 별도 변경 결정 전까지 구현이 따라야 하는 제한적 기본값
- **추후 검토**: 기본값은 채택했지만 세부 수치, transport 또는 UX를 다시 검토할 항목
- **미구현**: 설계만 존재하며 현재 제품 코드와 검증 근거가 없음

현재 이 문서의 모든 결정은 **기본 채택 · 추후 검토 · 미구현** 상태다.

## 비개발자를 위한 요약

1. 사용자가 입력한 원문은 Secure Gateway만 보고 Agent와 LLM에는 가린 내용만 보낸다.
2. 암호문 계산은 비밀키가 없는 Worker가 수행한다.
3. 결과를 열려면 PC와 스마트폰이 모두 참여해야 한다.
4. 최종 평문은 승인된 PC 또는 스마트폰 화면에서만 만든다.
5. 오류나 장치 부재가 발생하면 보안을 낮춰 계속하지 않고 중단한다.

전문 용어는 [`glossary.md`](glossary.md)에서 쉬운 설명을 확인할 수 있다.

## 신뢰 모델

| 영역 | 신뢰 | 설명 |
|---|---|---|
| 사용자와 trusted ingress | 신뢰 | 사용자 원문을 처음 받는 경로 |
| Secure Gateway | 신뢰 | pre-LLM masking과 post-LLM egress 강제 |
| Privacy Core | 신뢰 | 세션, handle, ciphertext와 policy 상태 소유 |
| Public Compute Worker | 기밀성 비신뢰, 초기 무결성 신뢰 | secret-free 공개 연산만 수행; 악성 Worker 검증은 후속 범위 |
| Reveal Coordinator | 신뢰 | reveal policy, destination, nonce와 partial decrypt protocol 조정; secret share 없음 |
| PC Partial Decrypt Authority | 최소 신뢰 | `sk_pc` 또는 exact-envelope unwrap share 보유; 단독 reveal 불가 |
| Phone Partial Decrypt Authority | 최소 신뢰 | `sk_phone` 또는 exact-envelope unwrap share 보유; 사용자 승인 후 참여 |
| Fusion Sink | 조건부 신뢰 | 두 partial share를 결합해 PC 또는 phone의 승인된 sink에서만 plaintext 생성 |
| OpenShell sandbox | 격리 경계 | 비신뢰 Agent의 파일, 프로세스, 네트워크 권한 제한 |
| Hermes Agent | 비신뢰 | 외부 Agent 구현체; 임의 tool 호출, handle 위조, 응답 조작 가능 |
| LLM/provider | 비신뢰 | masked prompt와 공개 정보만 허용 |
| PC OS/kernel | 조건부 신뢰 | 완전 침해 시 보호는 기본 보안 주장 밖 |
| Smartphone OS/app | 조건부 신뢰 | phone share와 phone-fusion plaintext 처리; 완전 침해 시 해당 장치 보호는 주장 밖 |

## 전역 불변식

1. 원문은 Secure Gateway 외부로 전달하지 않는다.
2. 비밀키와 plaintext 생성 권한은 Agent-facing 프로세스에 두지 않는다.
3. Agent는 raw ciphertext가 아니라 opaque handle을 기본으로 사용한다.
4. decrypt, unmask, reveal은 terminal local-egress에서만 실행한다.
5. 실패, 불확실성 또는 정책 불일치 시 전송·연산·reveal을 중단한다.
6. 권한은 요청자가 제출하는 `role` 문자열이 아니라 프로세스와 연결 채널로 결정한다.
7. 지원하지 않는 입력 형식과 데이터 흐름은 조용히 통과시키지 않는다.
8. OpenShell은 Agent 격리 계층이며 FHE-Privacy의 masking/reveal 정책을 대체하지 않는다.
9. Hermes와 Agent MCP Bridge는 하나의 비신뢰 sandbox principal이며 Bridge credential은 보안 경계가 아니다.
10. secure mode는 OpenShell connect/exec/sync/forward와 Agent 직접 입력을 차단한 sealed sandbox만 사용한다.
11. FHE key는 PC의 `sk_pc`와 스마트폰의 `sk_phone`으로 공동 생성하며 완성된 secret key를 생성,
    저장, 전송 또는 복구하지 않는다.
12. FHE reveal은 기본 2-of-2다. PC와 스마트폰 중 어느 한쪽 share만으로 plaintext를 만들 수 없다.
13. Exact secret은 record별 AEAD DEK로 암호화하고 DEK는 2-of-2 threshold envelope로 보호한다.
    단일 장치가 보유하는 공용 AEAD master key를 두지 않는다.
14. CKKS는 근사 수치, BFV/BGV는 범위가 정의된 exact integer, Boolean FHE는 exact predicate에만
    사용한다. 주민등록번호 같은 exact identifier는 기본적으로 threshold envelope에 저장한다.

## 명시적 초기 비범위

- RAG, vector DB, embedding과 document retrieval
- attachment, 이미지/OCR, 음성 입력
- 임의 workspace 또는 외부 memory의 자동 prompt 삽입
- 다중 사용자/원격 SaaS Privacy Core
- 네트워크 공개 MCP endpoint
- PC OS/kernel 완전 침해 방어
- production 보안 인증 또는 formal verification 주장

## 잔여 위험

다음 위험은 채택한 구조로 감소시키거나 영향 범위를 제한할 수 있지만 제거되지는 않는다.

| 잔여 위험 | 남는 이유 | 기본 대응과 주장 한계 |
|---|---|---|
| PII 미검출 | detector가 지원하지 않는 형식, 언어, 난독화 또는 새 PII를 놓칠 수 있음 | 지원 kind를 명시하고 ambiguity는 fail-closed한다. 모든 개인정보의 자동 검출을 주장하지 않는다. |
| OpenShell 탈출 | sandbox/runtime/kernel 취약점이나 잘못된 policy로 Hermes가 격리 경계를 벗어날 수 있음 | full-process 격리, hard filesystem requirement, 최소 network와 escape 회귀 테스트를 적용하되 완전한 격리를 주장하지 않는다. |
| 호스트 OS 침해 | 악성 kernel, 관리자 권한 공격자는 Gateway, Core, `sk_pc`와 PC-fusion plaintext를 관찰·변조할 수 있음 | 2-of-2로 PC 단독 복호를 막고 고위험 값은 phone fusion을 사용하지만 완전 침해된 host 보호는 비범위다. |
| 스마트폰 침해 | 악성 phone OS/app은 `sk_phone`, 승인 UI와 phone-fusion plaintext를 관찰·변조할 수 있음 | PC share 없이 단독 복호하지 못하게 하고 device binding을 적용하지만 완전 침해된 phone 보호는 비범위다. |
| 2-of-2 가용성 | PC 또는 스마트폰 분실·고장 시 reveal과 migration이 불가능함 | single-key fallback을 금지하고 별도 2-of-3 recovery 설계 전에는 fail-closed한다. |
| 공급망 공격 | Hermes/OpenShell/OpenFHE/Python dependency, image, wheel 또는 update가 오염될 수 있음 | pinning, checksum/signature, SBOM·출처 검증과 clean-build 검사를 적용하되 신뢰된 빌드 체인이 필요하다. |
| 화면·클립보드 탈취 | 정상 reveal 뒤 endpoint malware, screen capture, clipboard monitor가 plaintext를 가져갈 수 있음 | terminal sink 제한, clipboard 기본 금지, 고위험 phone-display-only를 사용하되 출력 이후 endpoint 침해는 막지 못한다. |
| Metadata/timing 노출 | marker 종류·개수, 길이, 반복 handle, 연산 패턴, 크기와 처리 시간이 남음 | 로그·metadata를 최소화하고 필요 시 padding/batching을 검토하지만 완전한 익명성이나 traffic-analysis 방어를 주장하지 않는다. |
| CKKS 오차 | CKKS는 본질적으로 근사 연산이며 depth, scale, encoding에 따라 오차가 누적됨 | 허용 오차가 정의된 numeric domain에만 사용하고 domain별 precision matrix를 검증한다. exact secret에는 사용하지 않는다. |
| Exact scheme 범위 오류 | BFV/BGV modular domain의 overflow나 잘못된 encoding은 정확하지만 의도와 다른 결과를 만들 수 있음 | modulus, signed encoding, 입력·중간 결과 범위와 overflow 거부 조건을 domain 계약으로 검증한다. |
| 구현 버그 | 검증, 직렬화, 암호 API, capability, cleanup 또는 fail-closed 로직에 결함이 생길 수 있음 | negative/fuzz/property/integration test, code review와 최소 권한을 적용하되 문서 설계만으로 correctness를 보장하지 않는다. |

따라서 목표 보안 주장은 “검증된 지원 경로에서 Hermes/LLM에 등록된 민감 plaintext와 secret key를
제공하지 않는다”로 한정한다. 위 잔여 위험까지 제거한 절대적 기밀성, 완전한 PII 검출 또는 host
compromise 방어를 주장하지 않는다.

## 검토 방법

각 문서는 제한적인 기본값을 먼저 채택한다. 추후 검토에서 완화하려면 다음을 함께 기록한다.

- 필요한 사용자 기능
- 새로 열리는 데이터 경로
- 공격자가 얻는 추가 capability
- 보완 통제
- negative/security test
- 보안 주장 문구의 변경

편의성만을 이유로 fail-closed 경로를 fail-open으로 바꾸지 않는다.
