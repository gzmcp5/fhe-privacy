# FHE-Privacy · OpenShell · Hermes 관계

> draw.io 원본: [`0. product-runtime-relationship.drawio`](0.%20product-runtime-relationship.drawio)
>
> 상태: 목표 설계 · 제품 구현 및 E2E 검증 전

## 핵심 관계

- 실행 관계: `사용자 → FHE-Privacy → OpenShell → Hermes`
- FHE-Privacy
  - 사용자가 설치·실행하는 최상위 개인정보 보호 제품
  - 개인정보 정책, 암호 상태, Vault와 reveal 경계 소유
  - OpenShell version·무결성 고정 및 실행 조정
- OpenShell
  - Hermes를 격리하는 별도 sandbox runtime
  - filesystem·process·network policy 강제
  - PII 정책, Vault, key share와 reveal 권한 미소유
- Hermes
  - sealed OpenShell sandbox 내부의 비신뢰 Agent
  - masked prompt와 opaque handle만 사용

## 주요 구성요소

- Secure Gateway
  - 사용자 raw input의 유일한 진입점
  - Agent/LLM 호출 전 PII masking
  - 최종 local egress 처리
- Privacy Core
  - session·handle·ciphertext·Vault metadata·policy 관리
  - Agent 요청의 context와 권한 검증
- Local Reveal
  - PC·스마트폰 2-of-2 참여 강제
  - 승인된 Fusion Sink에서만 plaintext 생성
- Stateless MCP Bridge
  - Hermes의 stdio MCP 요청을 agent-safe HTTPS/mTLS로 중계
  - 상태, key, Vault와 reveal 기능 미보유

## MCP와 MCP Bridge의 차이

| 구분 | 일반 MCP Server | FHE-Privacy MCP Bridge |
|---|---|---|
| 정체 | MCP 규격으로 tool을 제공하는 server | 기술적으로는 stdio MCP Server인 무상태 adapter |
| 기능 실행 | Tool 기능과 business logic을 직접 실행 가능 | 직접 실행하지 않고 Privacy Core로 중계 |
| 상태 | 자체 상태·저장소·credential 보유 가능 | 상태, Vault와 ciphertext 미보유 |
| 비밀정보 | 구현에 따라 key나 민감정보를 다룰 수 있음 | secret key/share와 reveal 기능 보유 금지 |
| 요청 변환 | 선택 사항 | MCP 요청을 agent-safe HTTPS/mTLS 요청으로 변환 |
| 응답 | 기능 실행 결과 반환 | Privacy Core의 result handle을 MCP 응답으로 변환 |
| 신뢰 수준 | 구현에 따라 결정 | Hermes와 동일한 비신뢰 영역 |
| 권한 검증 | Server가 직접 수행할 수 있음 | Privacy Core가 session·handle·operation 검증 |
| 호출 구조 | `Agent → MCP Server → 기능 직접 실행` | `Hermes → MCP Bridge → Privacy Core → 공개 연산` |

- MCP
  - Agent와 tool server 사이의 통신 규격
  - Tool 목록 조회, 호출과 응답 형식 정의
- 보안 기준
  - Bridge capability의 기밀성을 보안 근거로 사용하지 않음
  - Privacy Core의 agent-safe 권한 검증에 의존

## 데이터 경로

- 사용자 → Secure Gateway
  - raw input
- FHE-Privacy → OpenShell
  - 고정 version, artifact identity, sandbox policy, workload 설정
- Secure Gateway → Hermes
  - masked envelope, opaque handle, 최소 공개 metadata만 허용
- Hermes → MCP Bridge → Privacy Core
  - handle 기반 공개 연산 요청만 허용
- Hermes → Secure Gateway
  - unresolved response와 result handle만 허용
  - plaintext 반환 금지

## 정상 흐름

1. Secure Gateway가 사용자 원문을 수신하고 Privacy Core session을 시작한다.
2. Gateway가 PII를 검출·masking하고, 민감값을 용도에 맞게 암호화한 뒤 Vault에 등록한다.
   원문의 민감 구간은 opaque marker와 handle로 대체하며, 원문이 남아 있으면 요청을 중단한다.
3. Gateway는 `maskedText`, 내부 routing 정보, policy 정보로 구성된 masked envelope만 Hermes에 전달한다.
4. Hermes adapter는 envelope에서 `maskedText`와 허용된 공개 context·tool 설명만 골라 LLM 요청을
   구성한다. envelope 전체를 그대로 복사해 보내는 것은 아니다.
5. LLM은 masked text 응답을 만들거나, opaque handle을 참조하는 공개 연산 tool call을 Hermes에
   반환한다. 이때 LLM은 원문, ciphertext, Vault 위치, key를 알 수 없다.
6. 공개 연산이 필요하면 Hermes가 연산명, input handle, 공개 인자를 MCP Bridge에 전달한다.
7. MCP Bridge가 요청을 Privacy Core로 중계하면, Core는 session·handle type·출처·TTL·허용 연산을
   검증한다.
8. 검증에 성공한 경우 Core가 Vault에서 ciphertext를 조회해 Public Compute Worker에 전달한다.
   Worker는 공개 FHE context로 연산하고, Core는 결과 ciphertext를 Vault에 저장한 뒤 result handle을
   발급한다.
9. Hermes는 tool result인 result handle을 다시 LLM에 전달할 수 있다. 추가 연산이 필요하면
   5~9단계를 반복한다.
10. LLM이 최종 masked 응답을 반환하면 Hermes는 plaintext로 resolve하지 않은 unresolved response와
    result handle을 Gateway에 반환한다.
11. Gateway의 Reveal Coordinator가 result handle의 출처, reveal policy, 목적지, nonce를 검증한다.
12. 승인된 경우에만 PC·스마트폰이 partial decrypt 또는 unwrap에 참여하고, 승인된 Fusion Sink가
    이를 결합해 local-only plaintext를 생성한다.

### Hermes와 LLM 사이의 데이터

| 방향 | 전달 내용 | 전달하지 않는 내용 |
|---|---|---|
| Hermes → LLM | `maskedText`, 허용된 system/tool 설명, 필요한 최소 공개 context, opaque marker·handle | 사용자 원문, ciphertext, Vault 위치, key·share, reveal credential |
| LLM → Hermes | masked text 응답, handle 기반 tool call, 공개 연산 인자 | plaintext reveal 결과, secret 연산 자료 |
| Hermes → LLM (tool result) | MCP 공개 연산으로 발급된 result handle과 최소 상태 | 결과 ciphertext, 복호화된 값 |

- masked envelope는 Gateway와 Hermes 사이의 구조화된 내부 메시지다.
- LLM에는 envelope 자체가 아니라 Hermes adapter가 만든 provider 요청이 전달된다.
- `sessionId`, `messageId`, `policyVersion` 같은 내부 필드는 provider 호출에 꼭 필요한 경우에만 정책에
  따라 최소한으로 포함하며, 기본적으로 자동 전달하지 않는다.
- Hermes와 LLM의 왕복은 한 번으로 끝나지 않을 수 있다. LLM의 tool call과 result handle 반환을
  반복한 뒤 최종 unresolved response를 만든다.

## 보안 경계

- Hermes와 MCP Bridge
  - 동일한 비신뢰 principal
  - Bridge capability를 Hermes가 복제할 수 있다고 가정
- OpenShell
  - Agent 격리 담당
  - FHE-Privacy의 masking·Vault·reveal 정책을 대체하지 않음
- Agent-facing 영역에서 금지
  - raw plaintext와 key share
  - Vault filesystem과 host-only Core 접근
  - decrypt·unmask·key export·reveal
- 전체 시스템에서 금지
  - 완성된 secret key 생성·저장·전송
  - PC 또는 스마트폰 단독 reveal fallback
- 실패 처리
  - masking·암호화·Vault·격리·session·share 오류 시 fail-closed
