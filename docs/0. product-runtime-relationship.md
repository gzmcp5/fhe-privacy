# FHE-Privacy · OpenShell · Hermes 제품/런타임 관계

> 이 문서는 [`0. product-runtime-relationship.drawio`](0.%20product-runtime-relationship.drawio)의
> 박스, 경계와 화살표를 설명한다. 현재 내용은 목표 아키텍처이며 제품 구현이나 보안 검증 완료를
> 의미하지 않는다.

## 한 문장 요약

**FHE-Privacy가 사용자에게 제공되는 최상위 제품과 개인정보 보호 흐름을 소유하고, OpenShell은
Hermes를 격리하는 별도 sandbox runtime이며, Hermes는 그 sandbox 안에서 제한된 데이터와 도구만
사용하는 비신뢰 Agent다.**

```text
사용자
  -> FHE-Privacy Secure Gateway
       -> OpenShell Gateway / Runtime
            -> sealed sandbox
                 -> Hermes Agent + stateless MCP Bridge
```

이는 하나의 제품 사용 경험을 나타내지만 하나의 프로세스, 권한 영역 또는 container를 뜻하지 않는다.
FHE-Privacy, OpenShell runtime과 Hermes sandbox는 역할과 신뢰 수준이 서로 다른 별도 경계다.

## 세 시스템의 책임

| 시스템 | 위치와 신뢰 | 핵심 책임 | 소유하지 않는 것 |
|---|---|---|---|
| FHE-Privacy | 신뢰 호스트의 최상위 제품 | 사용자 진입점, PII masking, 암호 상태, Vault, 정책, local reveal | Agent sandbox 구현 자체 |
| OpenShell | FHE-Privacy가 버전과 무결성을 고정하는 별도 runtime dependency | sandbox lifecycle, filesystem/process/network 격리와 policy enforcement | PII 정책, Vault, key share, decrypt/reveal 정책 |
| Hermes | sealed OpenShell sandbox 내부의 비신뢰 Agent | masked prompt 처리, LLM 상호작용, 허용된 handle 기반 tool 호출 | raw input, secret key/share, Vault, plaintext reveal 권한 |

관계의 방향은 `FHE-Privacy가 OpenShell을 사용하고 OpenShell이 Hermes를 실행·격리한다`이다.
OpenShell이 FHE-Privacy를 포함하거나 Hermes가 FHE-Privacy의 내부 trusted component가 되는 구조가
아니다.

## FHE-Privacy 경계

다이어그램의 녹색 영역은 FHE-Privacy가 소유하는 신뢰 프라이버시 경계다. 개인정보 정책, 암호 상태,
key share 경계와 reveal 승인은 이 영역 밖의 Agent-facing 프로세스에 위임하지 않는다.

### Secure Gateway

- 사용자 raw input의 유일한 진입점이다.
- Hermes나 LLM을 호출하기 전에 PII 검출, masking, 암호화와 Vault 등록을 완료한다.
- 작업 중 하나라도 실패하거나 결과가 불확실하면 Agent 호출 전에 중단한다.
- Hermes가 반환한 unresolved response를 받은 뒤 local-only egress와 reveal 정책을 적용한다.

따라서 사용자가 Hermes의 stdin, TUI, webhook 또는 다른 직접 입력 경로로 원문을 전달하는 방식은
secure mode가 아니다.

### Privacy Core

- session, typed opaque handle, ciphertext, Vault metadata와 policy 상태를 소유한다.
- Agent가 요청한 handle의 session, type, context, provenance, TTL과 허용 operation을 검증한다.
- Agent-facing 요청에는 공개 연산만 허용하고 raw plaintext, decrypt, key export와 Vault 탐색을
  허용하지 않는다.
- Secure Gateway용 host-only interface와 sandbox용 agent-safe interface를 분리한다.

### Local Reveal 영역

- Reveal Coordinator, PC/Phone partial authority와 승인된 Fusion Sink로 구성한다.
- PC의 share와 스마트폰의 share가 모두 참여하는 2-of-2 경로만 허용한다.
- 완성된 secret key를 생성·저장·전송하거나 한 장치만 사용하는 fallback을 두지 않는다.
- 최종 plaintext는 승인된 PC 또는 phone Fusion Sink에서만 만들며 Hermes, MCP tool, LLM 또는
  conversation history로 반환하지 않는다.

## OpenShell 경계

다이어그램의 파란색 영역은 OpenShell runtime이다. FHE-Privacy 릴리스가 허용된 OpenShell version,
artifact checksum과 필요한 capability를 고정하며 설치·실행을 orchestration한다.

OpenShell Gateway/Runtime의 책임은 다음과 같다.

- sealed sandbox 생성과 종료
- sandbox policy 적용
- filesystem, process와 network 접근 제한
- 허용된 inference route와 agent-safe Core endpoint 외 egress 차단
- direct connect/SSH, exec, sync, forward와 service exposure 차단
- policy revision, workload identity와 runtime lifecycle 제공

OpenShell은 격리 계층이지 개인정보 보호 정책 엔진이 아니다. OpenShell이 정상 동작하더라도
FHE-Privacy의 pre-LLM masking, handle 검증과 2-of-2 reveal을 생략할 수 없다. 반대로 필요한
OpenShell 격리를 검증하지 못하면 FHE-Privacy secure mode도 시작하지 않는다.

## Sealed sandbox와 Hermes

주황색 점선 영역은 비신뢰 실행 영역이다. Hermes 전체 프로세스와 stateless MCP Bridge를 이 경계
안에서 실행한다.

### Hermes Agent

Hermes는 첫 reference Agent지만 trusted security component는 아니다. Hermes가 임의 tool 호출,
handle 위조, capability 복제, 응답 조작이나 직접 네트워크 접근을 시도할 수 있다고 가정한다.

Hermes에 허용되는 데이터는 다음으로 제한한다.

- masked prompt 또는 masked envelope
- opaque, typed, session-bound handle
- 공개 metadata와 허용된 operation 결과 handle
- 승인된 inference route의 LLM 응답

raw input, plaintext 민감값, secret share, Vault 경로, reveal credential과 host-only Core credential은
허용하지 않는다.

### Stateless MCP Bridge

- Hermes가 stdio MCP로 호출하는 얇은 protocol adapter다.
- MCP 요청을 agent-safe HTTPS/mTLS Core 요청으로 변환한다.
- session/Vault 상태, ciphertext 원본, secret key/share와 decrypt/reveal 코드를 보유하지 않는다.
- Hermes의 자식 프로세스이므로 Hermes와 같은 비신뢰 principal로 취급한다.

Bridge가 가진 agent capability는 Hermes가 복제할 수 있다고 가정한다. 보안은 capability 문자열의
기밀성이 아니라 Privacy Core가 agent-safe operation만 허용하는 데서 나와야 한다.

## 화살표와 데이터 의미

| 화살표 | 의미 | 허용 데이터 |
|---|---|---|
| 사용자 → Secure Gateway: `raw input` | 사용자 원문의 유일한 ingress | 지원되는 원문 입력 |
| FHE-Privacy → OpenShell: `pin · install · orchestrate` | runtime dependency의 버전 고정, 설치와 session 실행 | version, artifact identity, sandbox policy, workload 설정 |
| Secure Gateway → Hermes: `masked envelope only` | Agent/LLM 이전에 보호 처리가 끝난 입력 전달 | masked text, opaque handle, 최소 공개 metadata |
| Hermes → Stateless MCP Bridge: `stdio MCP` | Hermes의 허용된 tool 요청 | handle 기반 operation과 공개 인자 |
| MCP Bridge → Privacy Core: `handle-only HTTPS/mTLS` | 비신뢰 sandbox에서 agent-safe Core로 요청 중계 | session capability, handle, 허용 operation |
| Hermes → Secure Gateway: `unresolved response · no plaintext` | 최종 local egress 전 응답 반환 | masked text와 result handle; plaintext 금지 |

화살표가 없다는 것도 의미가 있다. Hermes 또는 MCP Bridge에서 Local Reveal, host-only Core, key share,
Vault filesystem으로 가는 직접 경로는 존재하지 않아야 한다.

## 정상 실행 흐름

1. 사용자는 FHE-Privacy CLI 또는 Secure Gateway를 통해 session을 시작한다.
2. FHE-Privacy는 고정된 OpenShell runtime과 policy로 sealed Hermes sandbox를 준비한다.
3. Secure Gateway가 raw input을 검출·마스킹하고 필요한 민감값을 암호화해 Privacy Core/Vault에
   등록한다.
4. Secure Gateway는 masked envelope만 Hermes에 전달한다.
5. Hermes는 masked prompt로 LLM과 상호작용하고 필요하면 opaque handle 기반 연산을 계획한다.
6. Hermes가 stateless MCP Bridge를 호출하면 Bridge가 agent-safe HTTPS/mTLS 요청으로 중계한다.
7. Privacy Core가 session과 정책을 검증하고 공개 연산 결과를 새 result handle로 저장한다.
8. Hermes는 plaintext가 없는 unresolved response와 result handle을 Secure Gateway에 반환한다.
9. Gateway egress가 reveal 가능 여부와 목적지를 검사한다.
10. 승인된 경우에만 PC와 phone partial authority가 참여하고 지정된 Fusion Sink에서 최종 plaintext를
    만든다.

## 반드시 유지할 경계

- FHE-Privacy 밖으로 raw plaintext, key share, Vault와 reveal 권한을 보내지 않는다.
- Hermes 전체와 MCP Bridge를 OpenShell sandbox 안에 둔다.
- Hermes와 Bridge를 별도 trusted principal로 간주하지 않는다.
- OpenShell Gateway, Secure Gateway, Privacy Core와 reveal component를 하나의 프로세스 권한으로
  합치지 않는다.
- Gateway를 우회하는 Agent 직접 입력을 secure mode에서 차단한다.
- sandbox에는 host home, key, Vault, profile과 reveal credential을 mount하지 않는다.
- agent-safe endpoint에서 raw plaintext/ciphertext, decrypt, unmask, resolve와 key export를 거부한다.
- 오류, ambiguity, 격리 실패, session mismatch 또는 share 누락 시 보안을 낮춰 계속하지 않고
  fail-closed한다.

## 설치와 배포 관계

사용자는 FHE-Privacy를 설치하고 실행한다. FHE-Privacy installer 또는 배포 과정이 검증된 OpenShell
runtime과 Hermes workload를 준비한다. 사용자가 secure mode에서 OpenShell CLI로 Hermes를 직접
실행하는 흐름은 제품 진입점이 아니다.

로컬과 Kubernetes 배포 모두 이 논리 경계를 유지한다.

- 로컬: FHE-Privacy host process, 별도 OpenShell runtime, sealed Hermes sandbox
- Kubernetes 후속 배포: FHE-Privacy component, OpenShell Gateway/Supervisor와 Hermes workload를
  별도 workload와 service account로 분리
- Windows 개발 경로: 현재 고정된 OpenShell 지원 방식에 따라 WSL 2 안에서 runtime을 실행하되,
  FHE-Privacy와 Hermes 사이의 데이터·권한 경계는 동일하게 유지

플랫폼별 설치 방식이 달라져도 `FHE-Privacy → OpenShell → Hermes`의 소유 관계와
`Secure Gateway ↔ Hermes ↔ MCP Bridge ↔ Privacy Core`의 제한된 데이터 경로는 바뀌지 않는다.

## 현재 상태와 주장 한계

이 다이어그램과 문서는 구현 계약의 상위 개요다. 현재 저장소는 제품 코드 재구축 전 설계 단계이며,
다음 항목을 아직 완료했다고 주장하지 않는다.

- FHE-Privacy 제품 package와 CLI
- Secure Gateway/Privacy Core 실제 프로세스
- Hermes adapter와 sealed OpenShell E2E
- PC/phone 장치 분리 2-of-2 reveal
- OpenShell filesystem/network/channel negative test
- production security 또는 모든 PII의 완전한 검출

OpenFHE wheel smoke test나 OpenShell binary의 checksum/version 확인은 backend 입력 검증 근거일 뿐,
이 전체 제품 관계와 보안 흐름이 구현·검증됐다는 근거가 아니다.
