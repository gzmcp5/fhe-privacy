# MCP Bridge and Stateful Privacy Core Boundary

이 문서는 RAG 없는 FHE-Privacy 아키텍처의 두 번째 확정 경계인 stdio MCP Bridge와
stateful Privacy Core의 프로세스/상태 배치를 정의한다.

## 결정

> OpenShell Agent가 실행하는 MCP stdio 프로세스는 무상태 Agent MCP Bridge다. 세션, marker,
> ciphertext와 key/context metadata는 신뢰 호스트의 session-scoped Privacy Core만 소유한다.
> FHE secret은 PC와 스마트폰의 2-of-2 share로만 존재하며 완성된 secret key는 어느 프로세스나
> 장치에도 존재하지 않는다.

- Gateway와 Agent는 같은 인메모리 MCP runtime을 공유하지 않는다.
- Hermes Agent와 Agent MCP Bridge는 모두 OpenShell sandbox 안에서 실행한다.
- Agent MCP Bridge에는 secret key, Vault 파일, plaintext 또는 reveal 기능을 두지 않는다.
- Agent MCP Bridge는 Hermes의 자식 프로세스이므로 별도 보안 주체로 신뢰하지 않는다. Bridge에
  전달된 capability는 Hermes를 포함한 sandbox의 임의 프로세스가 사용할 수 있다고 가정한다.
- Secure Gateway는 OpenShell Bridge를 거치지 않고 host-only Core interface를 사용한다.
- sandbox 전체는 agent-safe Core interface로 handle 기반 공개 연산만 요청할 수 있다. Bridge는
  stdio MCP와 Core protocol 사이를 변환하는 무상태 adapter일 뿐 권한 경계가 아니다.
- 초기 Privacy Core 수명은 Agent 대화 세션 단위로 둔다.
- Bridge가 재시작돼도 Core가 살아 있는 동안 세션 상태는 Core에 남는다.
- Core가 종료되면 초기 구현에서는 해당 세션을 fail-closed로 종료한다.

## 프로세스 배치

```text
Trusted host
  Secure Gateway
    -> host-only Core interface
    -> session-scoped Privacy Core
         - Session Registry
         - Handle/Ciphertext Vault
         - PII/Crypto ingress
         - Public FHE operation coordinator
    -> secret-free Public Compute Worker
    -> Reveal Coordinator (no secret share)
    -> PC Partial Decrypt Authority (sk_pc)
    -> Smartphone Partial Decrypt Authority (sk_phone)
    -> selected PC/phone Fusion Sink

OpenShell sandbox
  Hermes Agent (untrusted)
    -> MCP stdio
    -> stateless Agent MCP Bridge (untrusted adapter)
         -> HTTPS/mTLS agent-safe Core interface
```

Public Compute Worker, Reveal Coordinator와 PC Partial Decrypt Authority는 별도 프로세스와 권한
채널로 실행한다. Smartphone Partial Decrypt Authority는 별도 장치에서 실행한다. Core와 Worker는
어떤 secret share도 로드하지 않으며 완성된 secret key를 복구하는 API를 제공하지 않는다.

## 구성요소 책임

| 구성요소 | 상태 | 권한 |
|---|---|---|
| Secure Gateway | 현재 사용자 요청과 masked envelope | raw input 전달, masking/encryption 요청 |
| Agent MCP Bridge | 무상태, 비신뢰 | agent-safe handle 연산 중계; 독자 권한 없음 |
| Privacy Core | 세션, handle, ciphertext, key/context metadata | 상태 검증과 연산 dispatch |
| Session Handle Vault | marker/handle와 ciphertext의 관계 | Core 내부 접근만 허용 |
| Hermes Agent | 비신뢰 conversation/tool state | opaque handle 사용만 허용 |

## Agent MCP Bridge 제한

Bridge가 받을 수 있는 요청:

- 현재 세션에 속한 opaque handle
- 허용된 public FHE operation 이름
- 정책상 공개로 분류된 상수
- 요청 correlation ID

Bridge에 두지 않는 기능:

- raw plaintext 입력 암호화
- ciphertext 복호화
- marker/reveal resolve
- Vault 파일 직접 읽기 또는 쓰기
- key/context bundle export
- 다른 세션 handle 목록 조회
- local fallback 연산

Bridge는 요청 형식을 최소 검증한 뒤 Core에 중계한다. 실제 session, handle, type, context와 operation
권한 검증은 Core가 수행한다. Core에 연결할 수 없으면 MCP tool 오류를 반환하며 자체 상태나 키를
만들어 계속 실행하지 않는다.

## Core interface 분리

Privacy Core는 최소 두 개의 논리적 interface를 가진다.

| Interface | 호출자 | 허용 범위 |
|---|---|---|
| host-only Core interface | Secure Gateway | 세션 생성/종료, masking/encryption, 이후 local egress |
| agent-safe Core interface | OpenShell sandbox의 임의 프로세스 | handle 기반 공개 연산과 result handle 발급 |

호출자가 request field 또는 CLI의 `role` 값을 바꿔 권한을 얻는 구조는 사용하지 않는다. 권한은
연결된 interface와 session-scoped capability에 의해 결정한다. 단, capability의 기밀성은 권한
경계가 아니다. Hermes가 capability를 획득해 직접 호출해도 agent-safe 범위를 벗어나지 않아야 한다.

초기 transport는 다음으로 고정한다.

- host-only Core interface, Reveal Coordinator와 PC partial-decrypt interface는 서로 다른 local UDS에 bind한다.
- agent-safe Core interface만 HTTPS listener를 제공한다.
- Docker/Podman/VM에서는 `host.openshell.internal`의 전용 port를 사용하고, Kubernetes 지원 시에는
  전용 ClusterIP Service를 사용한다.
- TLS server identity를 검증하고, sandbox identity와 짧은 session capability를 함께 검증한다.
- capability는 `sandbox_id`, `session_id`, 허용 operation, expiry와 policy revision에 바인딩한다.
- OpenShell의 현재 gateway-to-sandbox SSH/TCP relay는 방향과 신뢰 모델이 다르므로 이 경로에
  재사용하지 않는다.

## 세션 수명주기

1. Gateway가 Privacy Core session을 시작한다.
2. Core가 session ID와 agent-safe capability를 만든다.
3. Gateway가 원문을 마스킹하고 Core Vault에 ciphertext를 저장한다.
4. Gateway가 masked envelope만 OpenShell Agent에 전달한다.
5. Agent가 stdio MCP Bridge에 handle 기반 연산을 요청한다.
6. Bridge가 agent-safe interface로 요청을 전달한다.
7. Core가 session, handle, context와 operation을 검증한다.
8. Core가 연산 결과 ciphertext를 저장하고 result handle을 반환한다.
9. Bridge는 result handle만 Agent에 반환한다.
10. Gateway가 세션을 종료하면 capability와 세션 상태를 폐기한다.

Hermes가 Bridge를 거치지 않고 같은 capability로 6단계를 직접 수행해도 Core의 결과는 같아야 한다.
Bridge 경유 여부나 실행 binary identity를 authorization 근거로 사용하지 않는다.

초기 구현에서는 Core 재시작 후 세션 복구를 지원하지 않는다. 복구되지 않은 handle을 추측해
재사용하는 대신 전체 세션을 종료하고 사용자가 새 세션을 시작하도록 한다.

## Fail-closed 규칙

| 상황 | 동작 |
|---|---|
| Core 연결 실패 | MCP tool 오류, fallback 금지 |
| Core session 없음 또는 종료됨 | 요청 거부 |
| 다른 세션 handle | 요청 거부 |
| 알 수 없는 handle | 요청 거부 |
| 다른 key/context ciphertext | 요청 거부 |
| 허용되지 않은 operation | 요청 거부 |
| 결과 저장 실패 | result handle을 반환하지 않음 |
| Bridge 재시작 후 capability 누락/불일치 | 요청 거부; 필요 시 Gateway가 살아 있는 세션 capability 재전달 |
| Core crash/restart | 초기 구현에서는 세션 종료 |
| quota 또는 resource limit 초과 | 요청 거부 |

## 초기 수명과 저장 정책

- Core는 세션 시작 시 생성하고 세션 종료 시 폐기한다.
- 세션 handle/ciphertext 상태는 초기에는 Core 메모리에 둔다.
- 장기 profile과 key material은 세션 Vault와 분리한다.
- 영속 Vault, crash recovery, key rotation은 별도 운영 설계 후 추가한다.
- Agent MCP Bridge 종료는 Core 상태 삭제 조건이 아니다.
- Gateway가 명시적으로 종료하거나 Core 자체가 종료될 때 세션을 폐기한다.

## 완료 검증

- Gateway와 Agent가 서로 다른 MCP/adapter 프로세스를 사용해도 같은 Core session handle을 처리한다.
- Agent MCP Bridge 프로세스에는 secret key와 Vault 파일이 없다.
- Hermes가 Bridge의 capability와 wire request를 복제해도 agent-safe operation 외 권한을 얻지 못한다.
- Bridge를 종료하고 다시 연결해도 살아 있는 Core session의 handle이 유효하다.
- Core를 종료하면 이전 handle 사용이 모두 거부된다.
- Agent가 host-only interface에 접근할 수 없다.
- 다른 세션 handle과 context가 다른 ciphertext 요청이 거부된다.
- Core 장애 시 Bridge가 plaintext 또는 로컬 ciphertext fallback을 만들지 않는다.
- Agent에는 raw ciphertext 대신 result handle만 반환하는 것을 기본으로 한다.

## 후속 설계와의 경계

구현 단계에서는 확정된 논리 경계를 실제 권한 경계로 강제한다.

- Agent, Gateway, Public Compute Worker, Reveal Coordinator, PC/phone partial authority와 Fusion Sink의
  프로세스와 권한 채널 분리
- agent-safe HTTPS/mTLS transport의 인증서 발급과 sandbox identity 결합
- session capability 발급, 인증, 짧은 만료, policy revision 결합과 폐기
- marker/handle의 타입, TTL, provenance와 연산 허용 목록
- host-only/coordinator/PC partial UDS의 filesystem permission과 peer identity
- smartphone enrollment, authenticated channel, partial share binding과 fusion destination
