# Cryptography and Data-Type Policy

## 기본 결정

CKKS는 근사 수치 연산에만 사용한다. 계산하지 않는 exact secret과 문자열 PII는 인증된 로컬
암호화 저장 방식으로 분리한다. 하나의 암호 방식으로 모든 PII를 처리하지 않는다.

## 데이터 분류

| 데이터 | 초기 정책 |
|---|---|
| 근사 허용 수치 | CKKS 후보 |
| exact count/integer | 오차 계약 전 FHE 지원 보류 |
| 회계·결제 금액 | 정확성 기준 확정 전 결과 주장 금지 |
| 이름, 전화번호, 주소 | AEAD 기반 local Vault 후보 |
| 비밀번호, API key, 인증코드 | AEAD + 기본 reveal 금지/step-up |
| 공개 상수 | 정책 검증 후 평문 연산 인자 허용 |

OpenFHE CKKS를 exact string 복원 저장소처럼 사용하지 않는다. Exact integer homomorphic operation이
필요하면 BFV/BGV 도입 또는 검증된 encoding/rounding 계약을 별도 결정한다.

## CKKS context 계약

각 ciphertext와 handle은 다음과 바인딩한다.

- scheme와 parameter set version
- key/context ID
- multiplicative depth
- scaling modulus와 precision target
- slot count와 value shape
- 입력 범위와 unit
- 허용 operation과 최대 연산 깊이
- 결과 rounding/display policy

다른 key/context, type 또는 shape의 ciphertext를 자동 변환하거나 혼합하지 않는다.

## Secret key 정책

- secret key는 Reveal Authority만 로드한다.
- Agent MCP Bridge와 Public Compute Worker에는 secret key를 두지 않는다.
- secret bundle을 일반 tool 또는 export API로 제공하지 않는다.
- 저장 중 key protection과 rotation metadata를 둔다.
- TPM/TEE는 선택적 hardening이며 endpoint 완전 침해 방어를 과장하지 않는다.
- threshold mode에서는 한 장치의 share만으로 plaintext를 만들 수 없어야 한다.

## 무결성과 입력 검증

- Vault metadata와 exact secret은 AEAD로 무결성을 보호한다.
- FHE ciphertext는 key/context/type metadata와 함께 검증한다.
- malformed serialization, 과대 payload, NaN/Inf와 범위 초과를 거부한다.
- operation depth와 precision budget을 초과하면 계산을 중단한다.
- 근사 결과에는 오차 허용 범위와 rounding 근거가 있어야 한다.

## 보안 주장 제한

- FHE는 계산 중 plaintext를 숨기지만 입력/출력 장치 침해를 막지 않는다.
- CKKS 결과는 exact equality를 보장한다고 주장하지 않는다.
- public/evaluation material은 복호 권한이 없지만 version과 integrity 관리는 필요하다.
- ciphertext size, operation pattern, marker type과 빈도 metadata leakage는 별도 잔여 위험이다.

## 추후 검토

- 초기 지원 numeric domain과 허용 오차
- exact integer용 scheme 또는 encoding
- AEAD algorithm과 master key 저장 방식
- OpenFHE parameter generation과 security level
- key rotation, context migration과 기존 ciphertext 처리
- threshold FHE backend spike
