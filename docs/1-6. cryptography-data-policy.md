# Cryptography and Data-Type Policy

## 쉽게 말하면

- 평균처럼 약간의 오차를 허용하는 계산은 CKKS를 사용한다.
- 개수처럼 정확해야 하는 정수 계산은 BFV/BGV를 검토한다.
- 같은지 확인하는 참/거짓 계산은 Boolean FHE를 검토한다.
- 주민등록번호처럼 계산할 필요가 없는 값은 일반 정확 암호화(AEAD)로 보관하되 PC와 스마트폰이
  모두 참여해야 열 수 있게 한다.

용어 설명: [`glossary.md`](glossary.md)

## 기본 결정

데이터의 표현 형식이 아니라 필요한 연산과 정확성 계약으로 cryptographic scheme을 선택한다.
CKKS는 근사 수치, BFV/BGV는 exact modular integer, Boolean FHE는 exact predicate에 사용한다.
계산하지 않는 exact secret과 문자열 PII는 record별 AEAD와 2-of-2 threshold envelope로 분리한다.

## 데이터 분류

| 데이터 | 초기 정책 |
|---|---|
| 근사 허용 수치와 통계 | CKKS |
| 범위가 정의된 exact count/integer | BFV/BGV 후보; modulus와 overflow 계약 필수 |
| exact equality, bitwise policy, 복잡한 조건식 | Boolean FHE 또는 검증된 BFV/BGV circuit 후보 |
| 회계·결제 금액 | exact domain과 overflow 기준 확정 전 결과 주장 금지 |
| 전화번호, 주소, 주민등록번호 | record별 AEAD + 2-of-2 threshold envelope |
| 이름 | 현재 제품 정책상 탐지·masking·Vault 저장 대상에서 제외 |
| 비밀번호, API key, 인증코드 | threshold envelope + 기본 reveal 금지 또는 phone-fusion only |
| 공개 상수 | 정책 검증 후 평문 연산 인자 허용 |

주민등록번호는 숫자가 아니라 leading zero, 자리와 checksum 의미를 가진 exact identifier다. 저장과
복원이 목적이면 FHE를 사용하지 않고 threshold envelope에 저장한다. 동일성 또는 exact predicate를
암호문 상태에서 계산해야 할 때만 고정 길이 digit/byte encoding과 BFV/BGV 또는 Boolean FHE를 사용한다.
생년월일처럼 필요한 파생 속성은 trusted ingress에서 검증 후 별도 typed value로 추출한다.

## FHE context 계약

각 ciphertext와 handle은 다음과 바인딩한다.

- scheme와 parameter set version
- key/context ID
- multiplicative depth
- scaling modulus와 precision target
- plaintext modulus, signed encoding과 overflow bound
- slot count와 value shape
- 입력 범위와 unit
- 허용 operation과 최대 연산 깊이
- 결과 rounding/display policy

CKKS handle에는 scale과 오차 예산을, BFV/BGV handle에는 plaintext modulus와 입력·중간 결과 범위를,
Boolean FHE handle에는 bit width, gate/bootstrapping budget과 predicate contract를 필수로 둔다. 서로 다른
scheme의 ciphertext는 명시적으로 검증된 scheme-switch protocol 없이는 자동 변환하거나 혼합하지 않는다.

다른 key/context, type 또는 shape의 ciphertext를 자동 변환하거나 혼합하지 않는다.

## Multiparty FHE key 정책

- PC와 스마트폰이 scheme/context별 2-of-2 multiparty key generation에 참여한다.
- PC는 `sk_pc`, 스마트폰은 `sk_phone`만 보유하고 완성된 secret key는 존재하지 않아야 한다.
- joint public key와 joint evaluation keys만 Gateway, Core와 Public Compute Worker에 배포한다.
- Agent MCP Bridge와 Public Compute Worker에는 secret share를 두지 않는다.
- secret bundle을 일반 tool 또는 export API로 제공하지 않는다.
- 저장 중 key protection과 rotation metadata를 둔다.
- TPM/TEE는 선택적 hardening이며 endpoint 완전 침해 방어를 과장하지 않는다.
- CKKS, BFV/BGV와 Boolean FHE의 keygen/evaluation-key/partial-decrypt 지원은 scheme별로 독립 검증한다.
- 어느 scheme도 검증 실패 시 single-key mode로 fallback하지 않는다.

## Exact secret 암호화 경로

Exact secret은 FHE ciphertext가 아니라 record별 AEAD ciphertext와 threshold-wrapped DEK로 저장한다.

1. Secure Gateway가 record별 random DEK를 생성해 exact plaintext를 AEAD로 암호화한다.
2. DEK는 PC와 스마트폰이 모두 참여해야 unwrap 가능한 2-of-2 threshold envelope로 보호한다.
3. Vault에는 AEAD ciphertext, nonce, associated-data hash, wrapped DEK, envelope/key-set version만 저장한다.
4. Gateway는 plaintext와 DEK를 Agent 호출 전에 폐기하고 장기 AEAD master key를 보유하지 않는다.
5. Reveal Coordinator가 승인한 요청에만 PC/phone unwrap share를 생성하고 지정된 fusion sink에서 DEK와
   plaintext를 복원한다.

Threshold envelope의 구체 primitive는 구현 전에 별도 spike로 확정한다. 단순히 같은 DEK를 PC와
스마트폰에 각각 저장하거나 완성된 master key를 백업하는 방식은 2-of-2로 인정하지 않는다.

## 무결성과 입력 검증

- Vault metadata와 exact secret은 AEAD로 무결성을 보호한다.
- FHE ciphertext는 key/context/type metadata와 함께 검증한다.
- malformed serialization, 과대 payload, NaN/Inf와 범위 초과를 거부한다.
- operation depth와 precision budget을 초과하면 계산을 중단한다.
- 근사 결과에는 오차 허용 범위와 rounding 근거가 있어야 한다.
- BFV/BGV는 모든 입력과 중간 결과가 정의된 modular domain 안에 있음을 검증하고 overflow를 거부한다.
- 주민등록번호는 trusted ingress에서 형식과 checksum을 검증하고 원본과 파생 속성을 별도 handle로 둔다.

## 보안 주장 제한

- FHE는 계산 중 plaintext를 숨기지만 입력/출력 장치 침해를 막지 않는다.
- CKKS 결과는 exact equality를 보장한다고 주장하지 않는다.
- BFV/BGV의 exact는 정의된 modular domain 안에서만 성립한다.
- Boolean FHE와 scheme switching은 backend와 parameter set별 검증 전에는 지원한다고 표시하지 않는다.
- public/evaluation material은 복호 권한이 없지만 version과 integrity 관리는 필요하다.
- ciphertext size, operation pattern, marker type과 빈도 metadata leakage는 별도 잔여 위험이다.
