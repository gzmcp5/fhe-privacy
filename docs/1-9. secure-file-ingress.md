# Secure File Ingress and Original File Storage

이 문서는 사용자가 prompt text 대신 파일을 제출할 때 Secure Gateway가 원본 bytes를 수신하고,
로컬 신뢰 영역에서 문서를 해석·마스킹해 비신뢰 Agent/LLM에 전달하는 경계를 정의한다. OCR은 제품
범위에서 제외한다. 현재 문서는 설계 결정이며 구현 또는 검증 완료를 의미하지 않는다.

## 결정

> 파일 원본은 Secure Gateway만 수신한다. Agent/LLM에는 원본 파일이나 일반 workspace 경로를
> 제공하지 않고, 신뢰 영역에서 완전성 검사를 마친 masked document projection과 opaque handle만
> 전달한다.

- 파일 bytes도 prompt 원문과 동일한 보호 대상 ingress다.
- 확장자가 아니라 magic bytes, container 구조와 허용 profile로 실제 형식을 판정한다.
- format별 parser는 네트워크와 일반 filesystem 접근이 없는 별도 격리 프로세스에서 실행한다.
- parser가 추출한 모든 text-bearing part를 Canonical Document IR로 정규화한 뒤 PII 정책을 적용한다.
- 일부 영역을 해석하지 못했거나 지원 범위가 불확실하면 문서 전체를 거부한다.
- OCR, 이미지 내용 분석과 이미지에서의 PII 검출은 지원하지 않는다.
- Hermes, 다른 MCP server, LLM provider의 attachment API와 workspace direct-read는 원본 ingress로
  사용하지 않는다.

## 지원 형식

### 초기 허용 목록

| Profile | 확장자 | 정규화 단위 | 필수 제한 |
|---|---|---|---|
| Plain text | `.txt` | line/paragraph | UTF-8, NUL 없음, 크기 제한 |
| Markdown | `.md`, `.markdown` | Markdown AST block/inline | raw HTML, include, local/remote asset 비활성화 |
| CSV | `.csv` | row/cell | UTF-8, dialect 확정, cell/row/column quota |
| TSV | `.tsv` | row/cell | UTF-8, 고정 tab delimiter, cell/row/column quota |
| JSON | `.json` | object/array/key/value | 중복 key, 비정상 수치, depth/size 초과 거부 |
| Word OOXML | `.docx` | part/section/block/table/cell | macro, OLE, embedded/external object 거부 |
| born-digital PDF | `.pdf` | page/block/span | 암호화, OCR 필요 page, active/embedded content 거부 |

TXT, Markdown, CSV, TSV와 JSON은 File Phase 1, DOCX와 born-digital PDF는 File Phase 2로 구현·검증한다.
형식이 초기 허용 목록에 있더라도 해당 phase와 format profile 검증이 완료되기 전에는 secure mode에서
받지 않는다.

### 후속 후보

| 형식 | 확장자 | 선행 설계 |
|---|---|---|
| Excel OOXML | `.xlsx` | hidden sheet/row/column, formula, comment, external link를 포함한 spreadsheet pipeline |
| PowerPoint OOXML | `.pptx` | note, hidden slide, master/layout text, relationship를 포함한 slide pipeline |
| OpenDocument | `.odt`, `.ods` | ZIP/XML part allowlist와 DOCX/spreadsheet 동등 coverage |
| YAML | `.yaml`, `.yml` | safe schema, tag/alias/depth 제한과 duplicate key 정책 |
| TOML | `.toml` | key/value IR과 중복/수치/날짜 정규화 정책 |
| source/config text | 별도 allowlist | UTF-8 plain-text profile과 language별 size/secret 정책 |

후속 후보는 명시적 설계와 negative test가 추가되기 전까지 비지원이다. XLSX와 PPTX는 일반 텍스트
파일로 취급하지 않고 각각 별도 구조화 문서 profile로 다룬다.

### 명시적 비지원

- 이미지, 스캔 PDF와 OCR이 필요한 문서
- legacy Office binary (`.doc`, `.xls`, `.ppt`)
- macro-enabled Office (`.docm`, `.xlsm`, `.pptm`)
- password-protected/encrypted PDF·Office 문서
- RTF, HTML, XML, EPUB
- ZIP과 일반 archive
- EML, MSG, MBOX 같은 email container
- executable과 형식을 판정할 수 없는 binary

비지원 형식을 plain text로 추측하거나 확장자만 바꿔 처리하지 않는다.

## Format별 coverage 계약

### PDF

허용 PDF는 모든 page object를 순회할 수 있는 born-digital 문서여야 한다.

| Page 상태 | 의미 | 처리 |
|---|---|---|
| `TEXT_COMPLETE` | text object를 완전히 추출 | 허용 |
| `TEXT_EMPTY` | 실제 빈 page이며 text/image content 없음 | 허용 |
| `IMAGE_ONLY` | 의미 있는 content가 image로만 존재 | OCR 필요로 거부 |
| `MIXED_UNVERIFIED` | text와 의미 있는 image content가 혼합 | 초기 profile에서 거부 |
| `PARSE_FAILED` | object 일부를 해석하지 못함 | 거부 |

암호화, embedded file, JavaScript, launch action, multimedia, 알 수 없는 object, page 또는 decompressed
size quota 초과도 거부한다. 후속 profile이 장식 이미지를 허용하더라도 이미지는 Agent/LLM과 결과
파일에 전달하지 않고, 분석 범위에서 제거됐음을 사용자에게 표시해야 한다.

### DOCX

본문, 표, header/footer, footnote/endnote, comment, hyperlink text와 URL, tracked change의 현재·삭제
값, text box, hidden text, document/custom property를 모두 검사한다. VBA, OLE, embedded package,
external template/image/relationship, 알 수 없는 text-bearing part와 손상된 OOXML package는 거부한다.

### Markdown

heading, paragraph, list, blockquote, table, fenced/inline code와 일반 link를 AST로 정규화한다. Raw HTML,
실행 가능한 URL, iframe, data URI, include/import, local file reference와 remote image/resource는 거부한다.
Link URL도 PII와 credential 검사의 대상이다.

### CSV, TSV와 JSON

CSV/TSV는 dialect와 encoding이 모호하면 거부하고 모든 header와 cell을 검사한다. JSON은 duplicate key를
거부하고 key와 scalar value를 모두 검사한다. Formula-like cell은 실행하지 않고 문자열 데이터로만
취급한다. Spreadsheet application으로 내보낼 때 발생하는 formula injection은 별도 egress escaping
정책으로 차단한다.

## 정상 ingress 흐름

```text
사용자 파일
  -> Secure File Ingress
  -> 크기/개수 quota, magic bytes, 실제 형식과 profile 검사
  -> chunk 단위 원본 AEAD 암호화 저장
  -> 격리된 format parser에 복호 stream 제공
  -> extraction coverage manifest + Canonical Document IR
  -> PII 검출, 암호화와 Vault registration
  -> masked document projection + opaque document handle
  -> OpenShell Hermes Agent / LLM
```

Gateway는 원본 전체를 평문 임시 파일로 먼저 기록하지 않는다. Parser가 random access를 요구하면
Vault Object Service가 인증된 seekable decrypting reader를 제공한다.

## Canonical Document Intermediate Representation (Document IR)

IR은 Intermediate Representation(중간 표현)의 약자다. Canonical Document IR은 서로 다른 파일
형식을 이후 PII 탐지와 정책 처리가 공통으로 사용할 수 있도록 변환한 표준 문서 중간 표현이다. 단순
연결 문자열이 아니라 원본 위치를 보존하는 구조화 표현이다.

```json
{
  "documentId": "opaque-document-id",
  "formatProfile": "docx-v1",
  "blocks": [
    {
      "blockId": "opaque-block-id",
      "kind": "paragraph",
      "location": {"part": "body", "index": 4},
      "segments": [
        {"kind": "text", "value": "연락처는 "},
        {"kind": "protected", "piiType": "phone", "handle": "opaque-handle"}
      ]
    }
  ],
  "coverage": {"status": "complete", "warnings": []}
}
```

- `documentId`, `blockId`와 PII handle은 예측 불가능하고 session-bound다.
- 원본 filename, host path, plaintext hash와 Vault object ID는 Agent에 노출하지 않는다.
- Parser coverage가 `complete`가 아니면 masked projection을 발급하지 않는다.
- Agent는 전체 또는 policy가 허용한 block projection만 읽으며 언제나 marker가 적용된 결과를 받는다.
- 문서 검색과 부분 읽기를 추가할 때도 tool result에 원문을 반환하지 않는다.

## 원본 파일 격리 저장

### 저장 위치와 권한

원본은 FHE-Privacy가 소유하는 OS별 private application-data root 아래에 저장한다. 프로젝트 workspace,
Hermes workspace, 다운로드·동기화 폴더, MCP resource directory와 cloud object storage에는 저장하지 않는다.

개념적 layout:

```text
private-data-root/
  vault/
    metadata.db
    objects/<random-prefix>/<random-object-id>.blob
    staging/
    quarantine/
```

경로는 OS API로 결정하며 시작 시 owner, permission/ACL, symlink/reparse point와 mount 경계를 검증한다.
Object filename에는 원본 filename이나 content hash를 사용하지 않는다.

### 암호화 object 형식

- 파일마다 cryptographically random DEK를 만든다.
- 원본은 bounded chunk 단위 AEAD로 즉시 암호화한다.
- associated data에 storage version, object ID, session ID, chunk index와 policy version을 결합한다.
- 인증된 manifest가 chunk count, 순서와 전체 ciphertext 구성을 고정한다.
- 원본 filename과 document property를 포함한 민감 metadata도 별도 AEAD record로 저장한다.
- DEK는 exact secret과 같은 2-of-2 threshold envelope로 보호하며 공용 single-device master key를 두지 않는다.
- 활성 세션에서 parser가 필요한 DEK는 최소 시간 동안 host의 제한된 memory에만 두고 disk, log, crash
  dump 또는 Agent-facing 프로세스에 기록하지 않는다.

Upload stream은 제한된 memory buffer로 형식 header를 검사하면서 동시에 encrypted object로 기록한다.
부분 upload, 인증 실패 또는 quota 초과 object는 유효 object로 commit하지 않는다.

### Parser 권한

Parser는 지정된 object의 read-only decrypting stream과 IR 출력 channel만 가진다. 네트워크, subprocess,
일반 workspace, 다른 Vault object, key/share 저장소와 Agent sandbox에는 접근하지 못한다. CPU, memory,
wall-clock, decompressed byte, recursion과 object count quota를 적용한다. Parser crash, timeout 또는 truncated
output은 문서 전체 거부로 처리한다.

### Metadata DB

평문 metadata는 random object ID, format/storage version, ciphertext size, state, 생성/만료 시각, session
binding, policy version과 parser 결과 code로 최소화한다. 원본 filename, title/author, 내부 path, 추출 text,
PII 위치와 document property는 암호화한다. 불필요한 content fingerprint는 저장하지 않으며 중복 검사가
필요하면 raw SHA-256 대신 local secret 기반 HMAC을 별도 정책으로 검토한다.

## 수명주기와 삭제

```text
RECEIVING -> ENCRYPTED -> PARSING -> MASKED -> ACTIVE -> EXPIRED -> DELETED
                          \-> QUARANTINED -> DELETED
```

- 중단된 upload는 commit하지 않고 즉시 정리한다.
- parser 실패 object는 진단용 plaintext 없이 제한된 quarantine TTL 뒤 삭제한다.
- 기본 session 파일은 session 종료 또는 TTL 만료 시 원본, IR, PII record와 projection을 정리한다.
- 영속 보관은 사용자의 명시적 선택과 별도 retention policy가 있을 때만 허용한다.
- SSD overwrite를 삭제 보장으로 주장하지 않는다. Ciphertext와 모든 DEK/envelope reference를 폐기하는
  cryptographic erasure를 기본 삭제 의미로 사용한다.
- Audit log에는 원문, filename, 추출 text, marker 원값과 복호 buffer를 기록하지 않는다.

## Fail-closed 규칙

| 실패 또는 불확실성 | 처리 |
|---|---|
| 확장자/MIME/magic bytes 불일치 | 거부 |
| 암호화 또는 password protection | 거부 |
| 비지원/active/embedded content | 거부 |
| OCR 필요 page 또는 image-only content | 거부 |
| parser warning, crash, timeout 또는 coverage 불완전 | 거부; Agent 호출 0회 |
| 원본 encrypted-object commit 실패 | 거부; handle 미발급 |
| IR 또는 PII/Vault registration 실패 | 거부; projection 미발급 |
| quota/decompression/recursion 초과 | 거부 |
| Agent의 direct file/path/provider attachment 요청 | 거부 |

## 구현 단계

1. **File Phase 1:** TXT, Markdown, CSV, TSV, JSON, encrypted object store, Canonical Document IR.
2. **File Phase 2:** DOCX와 born-digital PDF, coverage manifest, active/embedded-content 거부.
3. **File Phase 3:** XLSX, PPTX와 OpenDocument 후보를 각각 별도 profile로 검토한다.

OCR, 이미지 분석, legacy Office, archive와 email container는 위 phase에 포함하지 않는다.

## 완료 검증

- 지원 format별 정상 문서가 동일 구조의 masked IR을 만든다.
- PII canary가 Agent input, mock LLM request, parser log, audit log와 filename에 없다.
- hidden/comment/header/footer/tracked-change 등 format별 text-bearing part가 누락되지 않는다.
- malformed, encrypted, active, embedded, image-only와 mixed-unverified 문서에서 Agent 호출이 0회다.
- ZIP/decompression bomb, path traversal, external relationship, parser timeout과 crash가 격리된다.
- 평문 원본 임시 파일이 workspace, private store와 parser filesystem에 남지 않는다.
- Agent와 parser가 다른 object, Vault key/share, host path와 network에 접근하지 못한다.
- Session 종료 후 object/IR/handle이 무효화되고 ciphertext와 DEK reference가 정리된다.

## 보안 주장 제한

이 설계는 검증된 허용 profile에서 추출된 text를 Agent/LLM에 보내기 전에 마스킹하는 경계를 정의한다.
OCR이나 이미지 내용을 분석하지 않으며, 지원 profile 밖의 문서를 완전하게 검사한다고 주장하지 않는다.
Host OS/kernel 완전 침해, parser 공급망 침해와 모든 PII의 완전한 자동 검출은 별도 잔여 위험이다.

사용자가 결과를 파일로 받는 역방향 경계는
[`1-10. secure-file-egress.md`](1-10.%20secure-file-egress.md)를 따른다. 입력 원본 binary를 Agent가
수정하는 것이 아니라 Gateway가 검증한 Output Document IR로부터 새 파일을 로컬 렌더링한다.
